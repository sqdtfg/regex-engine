## 代码逻辑问题

### 严重（可能导致错误结果）

1. **posix_compat regexec 中 REG_NOSUB 标志误用**（posix_compat.c:145）
   - `prog->re_flags & REG_NOSUB` 检查的是编译时的标志，但 `regexec` 的 `eflags` 参数被完全忽略
   - 应检查 `eflags` 而非 `prog->re_flags`

2. **dfa_match 在 pos == input_len 时提前 break**（matcher.c:205）
   - `if (pos == input_len) break;` 在检查接受状态之后执行
   - 这导致如果最后一个字符到达接受状态，`pos == input_len` 时已经检查过接受状态了，逻辑正确
   - 但 `dfa_match_full` 的 `*p == '\0'` 检查在 `while` 循环外，如果输入为空字符串，`current_state` 始终是起始状态，需要起始状态本身就是接受状态才能匹配空串 — 这是正确的 POSIX 语义

### 中等（功能不完整或有潜在问题）

3. **capture.c 子组匹配策略效率低**
   - 每个捕获组都重新构建 NFA → DFA，对于嵌套组会产生指数级的重复构造
   - 建议：缓存子 AST 的 DFA 或使用 NFA 模拟

4. **posix_compat 中 nmatch 参数未充分利用**
   - regexec 的 pmatch 数组只填了第 0 组（完整匹配），nsub 是通过启发式统计的，但实际未填入 pmatch[1..nsub]

5. **main.c 未使用 api.h / posix_compat.h**
   - main.c 直接调用 dfa_match / dfa_dump 等底层 API
   - 应使用 regex_compile / regex_match 或 regcomp / regexec 替代

6. **capture.c 中 match_sub_dfa_greedy 的 last_accept 逻辑**
   - 当子表达式为空匹配（如 `(a)?` 匹配空串）时，last_accept 可能为 0，导致 `last_accept > text_start` 检查失败，返回 0 长度
   - 空匹配在 POSIX 中应返回 start=end 的位置

### 轻微（代码质量）

7. **matcher.c 中 format_range_label 的 is_dot 分支**：`lo >= 32 && lo <= 126` 包含了 `~`(0x7E)，但 126 是 `~`，127 是 DEL。范围正确。

8. **hopcroft.c 中 goto oom 模式**：资源清理分散，goto oom 后部分资源可能未释放（如 `work` 数组中的 transitions 在 oom 路径下可能未完全释放）

9. **tokenizer.c 中 parse_bracket 的 '-' 范围处理**：注释说"暂未展开，暂原样保留"，但实际 bracket_matches() 在 dfa.c 中实现了范围解析，parser 中只是把内容原样传给 NFA 构建

## DOT 标签显示问题

10. **复杂字符类标签可读性**：`[^a-z]` 显示为 `0x00-0x60,0x7b-0xff`，排除 26 个字符超出策略 3 的 5 个限制，无法语义化 — 这是已知限制

11. **`dfa_dump_dot` 中 `done_targets` 数组大小**：`int done_targets[256]` 用于标记已处理的 target 状态，但 DFA 状态数可能超过 256 — 应使用 `dfa->state_count` 或动态分配
