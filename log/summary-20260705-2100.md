# 2026-07-05 21:00 — 策略修正：精确区间匹配

## 修改的问题

1. 策略 4c 会把 `[0-35-9]` 误判为 `[0-9]`（区间覆盖了完整范围但中间有缺口）
2. 策略 4b 的 hex 数字检测过于宽松（`lo>=0x30 && hi<=0x39` 匹配了 `0-3` 等子范围）

## 解决的问题

1. **策略 4c 修正**：增加区间数量检查
   - 每个类别（数字/大写/小写）必须恰好 1 个区间且填满完整范围
   - `[A-MO-Z]` → `A-M,O-Z` ✅（不再是 `[A-Z]`）
   - `[0-35-9]` → `0-3,5-9` ✅（不再是 `[0-9]`）

2. **策略 4b 修正**：hex 数字区间精确匹配
   - `has_09` 现在要求 `lo==0x30 && hi==0x39`（完整 0-9）
   - `[0-9A-F]` → `[0-9-A-F]` ✅（精确 hex）
   - `[0-35-9A-F]` → `0-3,5-9,A-F` ✅（不走 hex，因为 0-9 不完整）

## 效果对比

| 模式 | 修复前 | 修复后 |
|------|--------|--------|
| `[0-35-9]` | `[0-9]` | `0-3,5-9` |
| `[A-MO-Z]` | `[A-Z]` | `A-M,O-Z` |
| `[0-9A-F]` | `[0-9-A-F]` | `[0-9-A-F]`（不变） |
| `[0-35-9A-F]` | `[0-9-A-F]` | `0-3,5-9,A-F` |

## 未解决的问题

1. 非标准子范围（如 `[0-9A-G]`）无法识别 — 这是合理限制
2. 复杂多区间字符类（如 `[a-zA-Z0-9]` 但缺少 `_`）无法还原为 `\w` — DFA 丢失语义

## 下一步计划

1. 考虑在 DFA 状态中附加原始 bracket 字符串，用于精确还原字符类标签
2. 或采用更激进的启发式：当排除字符恰好是连续可打印区间时，尝试推断原始模式
