# 2026-07-05 14:00 — matcher.c DOT 标签修复

## 修改的问题

1. `char_dot_label` 转义序列使用了 4 个反斜杠（如 `"\\\\n"`），应为 2 个（如 `"\\n"`）
2. DOT 通配符（`.`）显示为 `0x00-0xff`，视觉上混乱
3. 所有匹配式符号（`\d` `\D` `\w` `\W` `\s` `\S`）在 DOT 输出中显示为原始字节区间，缺乏语义化

## 解决的问题

1. **修复 `char_dot_label` 反斜杠数量**：`"\\\\n"` → `"\\n"`，`"\\\\r"` → `"\\r"`，`"\\\\t"` → `"\\t"`
2. **DOT 通配符识别**：新增 `semantic_range_label()` 函数，检测 `0x00-0xff` 全匹配 → 输出 `.`
3. **转义序列语义化**：`semantic_range_label()` 自动识别以下类别：
   - `.` — 所有 256 字节
   - `\d` — 0x30-0x39（数字）
   - `\D` — 非 0x30-0x39
   - `\w` — a-z A-Z 0-9 _
   - `\W` — 非 `\w`
   - `\s` — 空格、\t、\n、\r、\f、\v
   - `\S` — 非 `\s`
4. **负向转义序列修复**：`\D`、`\W`、`\S` 使用 `-2` 哨兵值区分"未见其他目标"和"其他目标为 -1（死状态）"

## 未解决的问题

1. 复杂字符类（如 `[^abc]`、`[a-zA-Z0-9]`）无法语义化识别，仍显示为原始字节区间 — 这是 DFA 层面丢失了语义信息的固有限制
2. `dfa_dump`（文本输出）未做语义化识别，仍显示原始区间 — 如需改进需单独处理

## 下一步计划

1. 考虑为 `dfa_dump` 也添加语义化识别（与 `dfa_dump_dot` 保持一致）
2. 考虑支持更复杂的字符类语义化（如 `[^abc]` → 可显示为 `!abc` 或类似）
