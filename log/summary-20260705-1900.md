# 2026-07-05 19:00 — hex 字符类 + 组合场景测试

## 修改的问题

1. `[0-9A-F]` 等 hex 子范围被策略 4b 近似为 `[0-9-A-Z]`，丢失了精度
2. 策略执行顺序不当 — 通用策略先于精确策略执行

## 解决的问题

1. **新增策略 4b：hex 字符类精确检测**
   - 在策略 4c（通用 ASCII 合并）之前执行
   - 精确检测 hex 子集：`[0-9]`、`[A-F]`、`[a-f]` 的任意组合
   - 输出：`[0-9A-F]` → `[0-9-A-F]`，`[0-9A-Fa-f]` → `[0-9-A-F-a-f]`

2. **策略执行顺序修正**
   - 精确策略（hex）在前，通用策略（a-z/A-Z/0-9）在后
   - hex 子集不会被误判为 full upper/lower case

3. **组合场景全面测试通过**
   - `\d{2,4}` → DOT 标签 `\d` ✅
   - `[a-zA-Z0-9]+\.[a-zA-Z0-9]+` → `[0-9-A-Z-a-z].[0-9-A-Z-a-z]` ✅
   - `(ab){2,3}` → `[a][b]` ✅
   - `\w+\s+` → `\w` + 原始区间 ✅

4. **`dfa_match_full` 精确匹配行为验证**
   - 代码已存在，从起始状态出发消耗全部输入后必须在接受状态
   - 与 `dfa_match`（子串匹配）语义不同，main.c 当前使用 `dfa_match`

## 效果对比

| 模式 | 修复前 | 修复后 |
|------|--------|--------|
| `[0-9A-F]` | `[0-9-A-Z]` | `[0-9-A-F]` |
| `[0-9A-Fa-f]` | `[0-9-A-Z-a-z]` | `[0-9-A-F-a-f]` |
| `[0-9a-f]` | `[0-9-a-z]` | `[0-9-a-f]` |
| `[a-zA-Z0-9]` | `[0-9-A-Z-a-z]` | `[0-9-A-Z-a-z]`（不变，非 hex） |

## 未解决的问题

1. `[0-9A-F]` 精确匹配为 hex，但 `[0-9A-G]` 等非标准子范围无法识别
2. `\w+\s+` 中 `\s` 在 DFA 层面有多个目标状态，无法被 `semantic_range_label` 识别为 `\s`

## 下一步计划

1. 支持更多常见子范围模式（如 hex 区间 [A-E]、[B-F] 等）
2. 或采用更激进的启发式：当排除字符恰好是连续可打印区间时，尝试推断原始模式
