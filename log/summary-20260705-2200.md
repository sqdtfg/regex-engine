# 2026-07-05 22:00 — 策略修正 + 补集描述修复

## 修改的问题

1. 策略 4c 会把 `[0-35-9]` 误判为 `[0-9]`（区间覆盖了完整范围但中间有缺口）
2. 策略 4b 的 hex 数字检测过于宽松
3. 策略 3 的排除区间收集逻辑有 bug — 遇到 `continue` 时不结束区间，导致区间被错误扩大

## 解决的问题

1. **策略 4c 修正**：每个类别必须恰好 1 个区间且填满完整范围
   - `[A-MO-Z]` → `A-M,O-Z` ✅（不再是 `[A-Z]`）
   - `[0-35-9]` → `0-3,5-9` ✅（不再是 `[0-9]`）

2. **策略 4b 修正**：hex 数字区间精确匹配 `lo==0x30 && hi==0x39`
   - `[0-9A-F]` → `[0-9-A-F]` ✅
   - `[0-35-9A-F]` → `0-3,5-9,A-F` ✅（不走 hex）

3. **策略 3 修复**：排除区间收集改用离散字符收集（≤5 个）
   - `[^abc]` → `[^abc]` ✅
   - `[^a-z]` → `0x00-0x60,0x7b-0xff` ✅（排除 26 个字符，超出限制）
   - `[^0-9]` → `\D` ✅（策略 1 语义化优先）

4. **删除重复的策略 4d** — 4c 和 4d 代码完全相同

## 未解决的问题

1. DFA 层面无法区分"排除 a,b,c"和"只匹配 d,e,f" — 两者都表现为 a,b,c 去死状态
2. `[^a-z]` 等大范围排除无法语义化（排除字符超过 5 个）
3. 非标准子范围（如 `[0-9A-G]`）无法精确还原

## 下一步计划

1. 考虑在 DFA 状态中附加原始 bracket 字符串，用于精确还原字符类标签
2. 或采用更激进的启发式：当排除字符恰好是连续可打印区间时，尝试推断原始模式
